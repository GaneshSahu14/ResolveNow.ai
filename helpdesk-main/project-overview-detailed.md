# ResolveNow AI — Comprehensive Project Documentation & Architecture Guide

ResolveNow AI is an enterprise-grade, AI-powered ticket management system. It automates support ticket ingestion, classification, resolution suggestion, and conversation summarization, while providing a clean dashboard for agents and administrators.

---

## 🗺️ Project Structure & Architecture

The project is structured as a **Monorepo** managed with **Bun Workspaces** (and configured for PNPM/Yarn if needed).

```
helpdesk-main/
├── core/             # Shared Types, Zod Schemas & Domain Constants
├── client/           # Vite + React + Tailwind + TypeScript Frontend
├── server/           # Express + TypeScript + Prisma + pg-boss Backend
└── e2e/              # Playwright End-to-End Test Suite
```

## 📦 1. Core Module (`/core`)
The `core` package acts as a shared library. It has `erasableSyntaxOnly` enabled, meaning it defines plain Typescript types, shared Zod schemas, and runtime constants (using `as const` objects instead of TypeScript `enum`s to keep build outputs clean).

### Key Subdirectories:
* **`core/schemas/`**: Shared validation schemas using Zod.
  * [`tickets.ts`](file:///d:/helpdesk-main/helpdesk-main/core/schemas/tickets.ts): Schemas for ticket creation, listing queries (with page/sort parameters), and inbound emails.
  * [`users.ts`](file:///d:/helpdesk-main/helpdesk-main/core/schemas/users.ts): Schemas for user registration, user update, and credentials validation.
  * [`replies.ts`](file:///d:/helpdesk-main/helpdesk-main/core/schemas/replies.ts): Schemas for ticket replies and reply polishing payloads.
* **`core/constants/`**: Shared constants used by both frontend and backend.
  * [`role.ts`](file:///d:/helpdesk-main/helpdesk-main/core/constants/role.ts): Defines roles (`admin` | `agent`).
  * [`ticket-status.ts`](file:///d:/helpdesk-main/helpdesk-main/core/constants/ticket-status.ts): Defines statuses (`new`, `processing`, `open`, `resolved`, `closed`).
  * [`ticket-category.ts`](file:///d:/helpdesk-main/helpdesk-main/core/constants/ticket-category.ts): Defines categories (`general_question`, `technical_question`, `refund_request`).

## 🗄️ 2. Database Schema (`server/prisma/schema.prisma`)
The system uses **PostgreSQL** configured via **Prisma ORM**. Below is the schema structure:

* **`User`**: Support agents and administrators. Features standard Better Auth parameters and a `role` field.
* **`Session` & `Account`**: Used by Better Auth to handle database-stored sessions and authentication.
* **`Ticket`**: Stores support tickets.
  * Statuses flow: `new` → `processing` → `open`/`resolved`/`closed`.
  * Can be assigned to a human `User` or left unassigned.
* **`Reply`**: Messages corresponding to a ticket thread. Can be created by a customer, agent, or generated by the AI system (where `userId` is `null`).
* **`Verification`**: Handles verification tokens for authentication workflows.

### Custom PostgreSQL Stored Procedures
The database includes custom functions like `get_ticket_stats(ai_agent_id)` used to calculate system-wide metrics (AI resolution rate, average resolution time, total and open ticket counts).

## ⚙️ 3. Backend Architecture (`/server`)
The server uses **Express 5** (which handles rejected promises in async routes automatically) running on **Bun**.

### Key Infrastructure:
1. **Authentication (Better Auth)**:
   * Mounted at `/api/auth/{*any}`.
   * Leverages a database-backed session model.
   * Protects endpoints using the `requireAuth` middleware, which populates `req.user` and `req.session`.
2. **Background Jobs (`pg-boss`)**:
   * Uses `pg-boss` (PostgreSQL-backed job queue) running in the `pgboss` schema.
   * `startQueue()` initializes the workers before the Express app starts listening.
   * Graceful shutdown hooks clean up queue listeners on `SIGTERM` / `SIGINT`.
3. **AI SDK (Vercel AI SDK + Groq)**:
   * Uses Groq (`Llama 3.1` model by default) to perform classification, drafting, and automated answering.

### Job Queues:
* **`classify-ticket`**:
  * Automatically triggers on new inbound tickets.
  * Uses the LLM to classify the ticket into one of the designated categories (`general_question`, `technical_question`, `refund_request`).
* **`auto-resolve-ticket`**:
  * Triggers on new inbound tickets.
  * Ingests the `server/knowledge-base.md` file.
  * Instructs the LLM to construct a personalized reply addressing the customer by their first name.
  * If the knowledge base contains insufficient information, the model returns exactly `"ESCALATE"`, which sets the ticket status to `open` (making it visible in the agent UI).
  * If the LLM answers successfully, the system creates a reply, marks the ticket as `resolved`, and enqueues an outbound email.
* **`send-email`**:
  * Sends outbound customer emails using Nodemailer.

### API Routes & Endpoints:
* **`/api/auth`**: Auth controller. Sign-up is disabled; users are managed via database seeding or admin controls.
* **`/api/me`**: Returns the current logged-in user profile.
* **`/api/users`**: CRUD operations for agents and administrators (restricted to admins).
* **`/api/agents`**: Returns list of active agents (used for assignment).
* **`/api/tickets`**:
  * `GET /`: Lists tickets with advanced filtering (status, category, search) and pagination. Excludes `new` and `processing` tickets unless explicitly requested.
  * `POST /`: Manual ticket creation.
  * `GET /:id`: Retrieves ticket details and assignment.
  * `PATCH /:id`: Updates ticket properties (status, category, assigned agent).
  * `GET /stats`: Computes dashboard metrics using the raw PostgreSQL query.
  * `GET /stats/daily-volume`: Aggregates ticket creation numbers over the last 30 days.
* **`/api/tickets/:ticketId/replies`**:
  * `GET /`: Lists thread replies.
  * `POST /`: Sends a reply and triggers an outbound email.
  * `POST /summarize`: AI summarizing endpoint. Returns a 2-4 sentence summary of the conversation thread.
  * `POST /polish`: AI polishing endpoint. Takes the draft, corrects grammar, improves tone, signs off with the agent's name, and appends a standard link footer.
  * `POST /suggest`: AI suggests a response based on the customer message, conversation history, and knowledge base.
* **`/api/webhooks/inbound-email`**:
  * Multiplexer webhook handling CloudMailin or SendGrid multipart payloads.
  * Strips subject prefixes like `Re:` or `Fwd:`.
  * If an existing open ticket exists for the sender with the same subject, it appends a reply. Otherwise, it creates a new ticket and schedules classification & auto-resolution jobs.

## 🎨 4. Frontend Architecture (`/client`)
Built using **React**, **Vite**, **TypeScript**, and styled with **Tailwind CSS** + **shadcn/ui**.

### Key Technology Choices:
* **State Management**: **TanStack React Query** (`useQuery` and `useMutation`) handles API interaction caching and invalidation instead of local state variables.
* **Forms**: **React Hook Form** paired with the Zod resolver (`@hookform/resolvers/zod`).
* **HTTP Requests**: **Axios** configuration with request/response interceptors.
* **Routing**: **React Router** with nested route configurations:
  * `ProtectedRoute` verifies session status and redirects unauthorized users to `/login`.
  * `AdminRoute` restricts views to administrators (redirects standard agents to dashboard).
* **Aesthetics**: Sleek design featuring custom shadcn semantic variables, dark mode styles, bento grids, card systems, and interactive UI states.

### Key Pages:
* **`DashboardPage.tsx`**: Displays statistics, volume charts (recharts), bento-grid widgets, and recent ticket activity.
* **`TicketsPage.tsx`**: Searchable and filterable ticket table with sorting, filters component, and pagination.
* **`TicketDetailPage.tsx`**: Chronological message thread layout (Intercom/Linear style), including:
  * Sidebar ticket metadata (Status, Category, Assignee).
  * Collapsible AI Assistant panel offering **Summarize**, **Suggest Reply**, and **Polish Draft** features.
* **`InboxPage.tsx`**: Linear/Slack-like unified ticket triage interface.
* **`KnowledgeBasePage.tsx`**: Interface to view and update the server's documentation and guides.
* **`AnalyticsPage.tsx`**: Detailed reports of volumes, response durations, and classification ratios.
* **`AIInsightsPage.tsx`**: Reports highlighting AI resolution performance, error rates, and automated cost savings.
* **`SettingsPage.tsx`**: Agent settings and theme customization.
* **`UsersPage.tsx`**: Admin portal to invite, update, and manage support agents.

## 🧪 5. Testing Framework
Testing is split into two components to ensure robustness without duplicating effort:

### Component Tests (`Vitest` + `React Testing Library`)
* Focused on UI rendering, element display, validation logic, and component states.
* Located alongside their components in the `client/src` directory (e.g. `TicketDetailPage.test.tsx`).
* Utilizes a custom `renderWithQuery` wrapper to mock queries/mutations.
* Axios is mocked using Vitest mocks (`vi.mock("axios")`).

### E2E Tests (`Playwright`)
* Located under `/e2e`.
* Reserved for testing complete workflows requiring both client and server:
  * Authentication flows and redirects.
  * Inbound webhook integration (webhook payload ingestion testing).
  * Multi-page routing and complex persistence checks.

## 🔑 6. Configuration & Environment Variables

### Server Env Variables (`server/.env`):
| Variable | Required | Description |
|---|---|---|
| `DATABASE_URL` | Yes | PostgreSQL connection string |
| `BETTER_AUTH_SECRET` | Yes | Token hashing secret (Base64) |
| `BETTER_AUTH_URL` | Yes | Public server URL (e.g. `http://localhost:3000`) |
| `TRUSTED_ORIGINS` | Yes | Cors origin allowed (e.g. `http://localhost:5173`) |
| `GROQ_API_KEY` | Yes | Groq LLM API Key |
| `WEBHOOK_SECRET` | Yes | Secret validation token for inbound email webhook |
| `SMTP_HOST` / `SMTP_PORT` | Yes | Outbound mail credentials |
| `SMTP_USER` / `SMTP_PASSWORD` | Yes | Outbound mail authentication |
| `SMTP_FROM_EMAIL` | Yes | System outbound sender email |
| `SEED_ADMIN_EMAIL` / `SEED_ADMIN_PASSWORD` | Yes | Initial seed administrator login |
| `SENTRY_DSN` | No | Error tracking DSN |

### Client Env Variables (`client/.env`):
| Variable | Required | Description |
|---|---|---|
| `VITE_API_URL` | No | Targets backend URL (defaults to `http://localhost:3000` via dev proxy) |

## 🚀 7. Production Deployment (Single-Container)
In production, the application runs on a single container (such as **Railway**).
1. The Vite client is built into static assets inside `/client/dist`.
2. The Express server serves the static folder contents (`express.static`) and falls back to `index.html` for SPA routing.
3. Seeding commands are run on initial deploy using `bun run --cwd server prisma db seed` to establish the admin account.
